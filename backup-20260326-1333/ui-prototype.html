TCP 三次握手

⚠️ 异常场景：第二次握手丢包
模拟服务端发出 SYN-ACK 但客户端未收到的情况，观察 TCP 如何通过超时重传机制恢复连接。
1️⃣
客户端发送 SYN → 服务端正常收到
SYN=1, seq=100  ｜  客户端进入 SYN_SENT
✂️
服务端回复 SYN-ACK ← 网络丢包！
SYN=1, ACK=1, seq=300, ack=101  ｜  报文在途中丢失，客户端收不到
⏱️
客户端等待超时（约 1s），触发超时重传
RTO（Retransmission Timeout）到期，客户端重新发送 SYN=1, seq=100
🔁
服务端再次回复 SYN-ACK
服务端仍处于 SYN_RCVD，收到重传 SYN 后重新回复，流程继续
✅
客户端发送 ACK，连接最终建立
若重传次数超过 SYN_RETRIES（默认 6 次）仍失败，连接彻底放弃，报 ETIMEDOUT

当前步骤解读
客户端随机选择初始序列号（ISN=100），发送 SYN 报文向服务端请求建立连接。此时客户端进入 SYN_SENT 状态，等待服务端响应。
第 1 步 / 共 3 步 TCP 三次握手
🔧 自定义参数 客户端 ISN = 服务端 ISN = 修改 seq 值后点"应用"，动画标签和字段表将自动推导更新
✓
第 1 步：客户端发送 SYN
客户端向服务端发送 SYN 报文，请求建立连接。选择随机初始序列号 ISN=100，进入 SYN_SENT 状态。
SYN=1, seq=100
2
第 2 步：服务端回复 SYN-ACK
服务端收到 SYN 后，回复 SYN-ACK。确认客户端序列号（ack=101），同时选择自己的初始序列号（seq=300），进入 SYN_RCVD 状态。
SYN=1, ACK=1, seq=300, ack=101
3
第 3 步：客户端发送 ACK
客户端发送最后一个 ACK，确认服务端序列号（ack=301）。双方进入 ESTABLISHED 状态，连接建立完成。
ACK=1, seq=101, ack=301
💡 为什么是三次？
两次握手无法防止历史重复连接的初始化问题：若客户端旧 SYN 报文延迟到达服务端，服务端会误以为是新连接。第三次握手让客户端确认服务端序列号，从根本上解决此问题。
🤔 思考一下
seq=100 这个值是怎么来的？为什么不从 0 开始？
seq 是随机选择的初始序列号（ISN, Initial Sequence Number）。不用 0 是为了安全：若每次都从 0 开始，攻击者可以预测序列号，伪造 TCP 报文注入数据（TCP 序列号预测攻击）。随机 ISN 让每条连接的序列号空间都不同，大幅提升安全性。
